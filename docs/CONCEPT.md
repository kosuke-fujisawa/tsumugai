# CONCEPT - tsumugai

## tsumugai とは何か

tsumugai は、Markdown ベースで記述されたノベルゲームシナリオを、

> 実行前に意味を確定し、検証し、再利用可能にする

ための意味実行基盤、semantic runtime です。

単なるノベルゲームエンジンではありません。単なる DSL でもありません。

tsumugai は、シナリオを書く、理解する、検証する、実行する、他環境へ移植する、という一連の行為を「意味が追跡可能な構造」として扱います。

## 背景

従来のノベルゲーム制作では、多くの場合、独自 DSL、実行時解釈、エンジン依存、人手によるテストプレイによってシナリオが管理されてきました。

この方法は小規模では機能します。しかし、分岐や条件が増えるにつれて、次の問題を抱えやすくなります。

- 分岐や条件が把握しづらくなる
- 到達不能ルートや条件矛盾が埋もれる
- 実行しないと意味が分からない
- 他エンジンへの移植が難しい
- LLM が生成したシナリオの整合性を確認しづらい
- シナリオがブラックボックス化する

tsumugai は、この問題に対して「シナリオを意味として先に固定する」という方向からアプローチします。

## 基本思想

### 1. 実行前に意味を確定する

runtime は Markdown を逐次解釈しません。

Markdown の解釈は parser と compiler の段階で完了し、runtime は意味が確定した IR を進めるだけです。

```text
Markdown
  -> AST
  -> IR
  -> runtime
```

実行時には、ラベル解決、分岐構造解析、条件構造解析をできるだけ行いません。これにより、実行結果の決定性と検証可能性を高めます。

### 2. parser と runtime を分離する

tsumugai において、parser は「読む」、runtime は「進める」責務を持ちます。

runtime は Markdown を知りません。parser は実行状態を持ちません。

この分離により、同じシナリオ構造を通常実行、CUI 再生、静的解析、ドライラン、外部アダプターへ再利用できます。

### 3. 表示と意味を分離する

tsumugai は、描画、音声再生、UI、アニメーションを担当しません。

代わりに、`Emit(Event)` のような意味イベントを出力します。

```text
tsumugai: 何が起こるかを決める
adapter:  どう見せるかを決める
```

Event は Bevy、Swift、Web、CUI などのアダプターが解釈します。tsumugai-core は環境固有の表示実装に依存しません。

### 4. シナリオを読める構造にする

tsumugai の価値は、単に複雑な分岐を扱えることではありません。

状態、条件、差分、実行順序を、人間が追跡可能な形にすることを重視します。

そのため、CUI プレイヤー、実行ログ、analyzer、到達性検査、将来的なドライランを重要な機能として扱います。

### 5. 状態を明示する

状態は暗黙にしません。

フラグ、変数、進行位置、乱数 seed などは `State` として明示され、runtime は `State` と `Input` から次の `State` と `Output` を決定します。

同じ Markdown、同じ State、同じ Input は、常に同じ Output と次 State を返すことを目指します。ランダム性を使う場合も seed によって再現可能にします。

## 分岐に対する考え方

tsumugai は「分岐が多いこと」自体を価値とは見なしません。

重要なのは、差分に意味があることです。

そのため、無意味な分岐爆発やコンプリート不能な列挙ではなく、状態依存の変化、文脈の変化、関係性の蓄積、解釈の変化のような意味の差分を安全に扱える構造を目指します。

## 検証と修正支援

tsumugai は問題を検出するだけでなく、可能な範囲で「どこを、なぜ、どう直せばよいか」を返すことを目指します。

将来的な Diagnostic は、次の情報を持つ想定です。

- severity
- message
- span
- related spans
- suggestion
- rule id

これは人間の修正作業だけでなく、LLM にシナリオを再修正させるための構造化フィードバックとしても利用できます。

## Dry Run とエンディング検証

tsumugai は、ユーザー操作を伴わずにシナリオを探索できるドライラン機能を中核機能として扱います。

Dry Run は、次の検証を行う想定です。

- 全分岐の探索
- 到達可能なエンディングの確認
- 到達不能ルートの検出
- 無限ループまたは最大深度超過の検出
- エンディングに到達しないルートの検出
- 条件によって選べない選択肢の検出

そのため、エンディングは単なるファイル末尾ではなく、明示的な意味単位として扱う方針です。

```markdown
[END id=good]
```

または

```markdown
[ENDING id=good name=グッドエンド]
```

のような DSL を検討します。

## LLM との関係

tsumugai は LLM 時代を前提とします。

そのため、Markdown ベース、人間可読、構造が単純、検証可能、ログ可能であることを重視します。

目的は LLM にシナリオを書かせることだけではありません。生成物を検証し、安全に扱えるようにすることです。

LLM は分岐、ラベル、条件、エンディング整合性を崩しやすいので、tsumugai は生成物を検証する決定的なコアとして機能します。

## CUI Player の意味

CUI Player は単なるデバッグツールではありません。

これは参照実装、意味確認環境、シナリオ検証環境、最小実行系です。

ここで、Enter で進む、Choice で止まる、Undo できる、という最小体験を保証します。

## アーキテクチャ思想

tsumugai は、形式的な Clean Architecture や深いオブジェクト階層そのものを目的にしません。

必要な境界は保ちつつ、次の性質を優先します。

- 明示的状態
- enum 中心
- 純関数
- 単純な責務分離
- 決定的な実行
- 小さく安定した外部契約

外部契約として特に重要なのは、`Event`、`State`、`Input`、`Output`、`Diagnostic`、`DryRunReport` です。これらはアダプターが依存する境界であり、後方互換性を意識して変更します。

## Adapter 境界

tsumugai-core は、iOS、Bevy、Web などの具体的な表示・再生実装を直接持たなくてもよいです。

各環境向けの変換層は、別 crate または別リポジトリで提供できます。

```text
tsumugai-core
  parser / analyzer / runtime / dryrun

tsumugai-cli
  check / play / dry-run / dump

tsumugai-bevy-adapter
  Event -> Bevy ECS / Asset / State

tsumugai-ios-adapter
  Event -> Swift DTO / UI layer

tsumugai-editor-support
  diagnostics / suggestions / JSON output
```

## 将来像

tsumugai は、単なる個人用ツールではなく、ノベルゲーム制作における意味の共通基盤を目指します。

具体的には、マルチエンジン対応、マルチプラットフォーム、既存 DSL からの移行、LLM 生成物検証、長編シナリオ管理を視野に入れます。

## 一言で言うと

tsumugai は、ノベルゲームシナリオを「実行されるテキスト」ではなく、「検証可能な意味構造」として扱うための基盤です。
